3.3 — ST Edge AI Core

The Python-to-C Bridge

ST Edge AI Core serves as the critical compiler bridging the machine learning environment and the embedded target. It ingests quantized neural network topologies (.tflite, .onnx) to generate hardware-optimized C source arrays. During this translation, the compiler schedules operational execution across the Neural-ART NPU and the host CPU, outputting a comprehensive epoch-level profiling report.

Compiler Architecture and Functional Scope

A serialized neural network topology—even a heavily optimized, quantized INT8 .tflite file—cannot be executed natively on bare-metal microcontroller architectures. These portable formats require ahead-of-time (AOT) compilation to translate generic graph definitions into hardware-specific execution schedules. ST Edge AI Core functions as this dedicated AOT compiler.

It does four things in one pass:

1. Graph optimization

Executes graph-level optimizations by fusing adjacent algorithmic layers (e.g., condensing Conv2D + BatchNorm + ReLU into a single monolithic kernel), eliminating redundant tensor transposes and formatting memory layouts specifically for the target NPU.

2. NPU epoch assignment

Analyzes the computational graph to partition operations. Supported layers are mapped to an EC (Execution Context) epoch for hardware acceleration on the Neural-ART NPU, while unsupported operators are routed to a SW (Software) epoch for fallback execution on the Cortex-M55 CPU.

3. Memory allocation

Maps static model weights to external memory (OctoFlash) and orchestrates activation buffer placement across internal SRAM domains (e.g., npuRAM4, npuRAM5) in strict adherence to the STM32N6570-DK .mpool memory topology configuration.

4. C code generation

Synthesizes the final execution environment, producing the runtime scheduling logic (network.c), the serialized weight arrays (network_data_params.c) and associated configuration headers. These artifacts are immediately ready for cross-compilation within STM32CubeIDE.

Compiler Invocation and Parameter Analysis

While the stedgeai command-line interface is typically abstracted and invoked automatically by the common_deploy.py orchestration script, analyzing the underlying command provides critical insight into the compilation constraints. The exact instruction set generated and executed for the MoveNet deployment is detailed below:

stedgeai generate \ --target stm32n6 \ -m st_movenet_lightning_heatmaps_192_int8_pc.tflite \ --st-neural-art default@user_neuralart_STM32N6570-DK.json \ --input-data-type uint8 \ --inputs-ch-position chlast \ --load-mpool stm32n6-app2_STM32N6570-DK.mpool \ --output experiments_outputs/2025_05_27_14_46_07/ \ -O3 --Oalt-sched --Ocache-opt --Os \ --enable-epoch-controller
Compilation Flag Architectural Function
--target stm32n6 Specifies the STM32N6 Neural-ART NPU as the primary hardware compilation backend.
--st-neural-art Ingests the target-specific hardware JSON profile for the STM32N6570-DK board.
--load-mpool Defines the hardware-specific memory topology, strictly mapping buffers to defined SRAM banks and absolute addresses.
--input-data-type uint8 Forces input mapping to uint8, matching raw camera frame buffers and eliminating the need for runtime CPU dequantization.
--enable-epoch-controller Activates the generation of the runtime epoch controller, responsible for orchestrating the interleaved execution of NPU (EC) and CPU (SW) epochs.
-O3 --Oalt-sched --Ocache-opt Enables maximum graph optimization (Level 3), alternative execution scheduling, cache-optimized memory layouts, and binary size reduction (-Os).

While this demonstrates highly deterministic and efficient real-time execution, the differential between effective throughput and the NPU's 600 GOPS theoretical peak underscores the latency bounds introduced by physical memory bandwidth constraints.

Internal Architecture of network.c

The network.c artifact consists of auto-generated C source code orchestrating the model's execution. Here we analayze its tripartite structure is fundamental to understanding the low-level hardware orchestration performed by the NPU at runtime. This analysis transitions the deployment process from an opaque script execution to a transparent architectural implementation.

Part 1 — Memory Pool Allocation Heuristics

The initial block encapsulates the compiler's memory allocation telemetry. Each target memory pool is evaluated against a heuristic score—derived from bandwidth, latency and cacheability metrics. The compiler maps each tensor to the highest-scoring valid memory domain:

/* global pool 8 is 2.92 MB */
/* name=octoFlash offset=0x70380000 READ_ONLY LATENCY=HIGH cacheable=ON score=50 */

/* global pool 1 is 432.00 KB */
/* name=npuRAM5 offset=0x342e0000 THROUGHPUT=HIGH LATENCY=LOW cacheable=OFF score=94 */

/* global pool 2 is 378.00 KB */
/* name=npuRAM4 offset=0x34270000 THROUGHPUT=HIGH LATENCY=LOW cacheable=OFF score=94 */

The internal npuRAM4 and npuRAM5 banks achieve a maximum score of 94, reflecting their zero-wait-state, direct NPU connectivity. Conversely, the external octoFlash scores 50 due to inherent latency constraints, yet remains the sole valid target for the 2.9 MB static weight payload.

Part 2 — Hardware Execution Context (EC) Blobs

A defining architectural feature of the Neural-ART NPU is its execution paradigm. The NPU does not process standard C function calls; instead, each EC epoch is encoded as a predefined binary blob. These 64-bit instruction words function as microcode, ingested directly by the hardware Epoch Controller:

/* Epoch 1 — first Conv2D block, MobileNetV2 stem */
static const uint64_t _ec_blob_1 [] =
{
  0x0000b686ca057a7aUL, /* CONVACC unit config */
  0x5c00004200802241UL, /* weight address + stride */
  0x0001b000342e0000UL, /* activation buffer (npuRAM5) */
  ... /* 71 blobs total, one per EC epoch */
};

Each blob encapsulates the absolute hardware state required for a single NPU epoch, including MAC unit configuration, OctoFlash weight pointer addresses, SRAM activation buffer limits and convolutional stride/padding parameters. By loading these directly into the NPU registers without CPU intervention, the architecture minimizes control-flow overhead and maximizes inference throughput.

Part 3 — Software (SW) Fallback Routines

For operations mapped to SW epochs, the compiler synthesizes standard C function invocations targeting the ll_sw (Low-Level Software) library. These are executed by the Cortex-M55 host CPU, accelerating vector math via the Helium SIMD extension (mandated by the --mvei compiler flag):

/* Epochs 59, 63, 67 — Resize bilinear (decoder upsample) */
ll_sw_resize_bilinear(input_ptr, output_ptr,
  in_h, in_w, out_h, out_w, channels);

/* Epoch 74 — DequantizeLinear (INT8 → float32 for postprocessor) */
ll_sw_dequantize(int8_ptr, float32_ptr, scale, zero_point, size);

Following the execution of the final SW epoch (epoch 74), the terminal output is an FP32 heatmap tensor localized in npuRAM5. This tensor is immediately accessible to the bare-metal C post-processing routines (e.g., display_spe.c) for coordinate decoding and spatial rendering.

Firmware Integration Context: The network.c artifact is volatile and excluded from manual version control. The custom C firmware interfaces with this auto-generated logic exclusively by invoking LL_ATON_RT_Main() within main.c. This low-level API call initializes the Neural-ART Epoch Controller, triggering the sequential execution of all 75 defined epochs. The only generated header directly included within the firmware is app_config.h.

Configuration Interface: app_config.h

In parallel with the executable C sources, the compilation pipeline synthesizes app_config.h via the gen_h_file.py utility. This header serves as the configuration bridge, providing the bare-metal firmware with the requisite topological and post-processing hyperparameters mapped from the original model.
