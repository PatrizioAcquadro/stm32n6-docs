3.4 — STM32CubeIDE & C Firmware

Cross-Compilation, Secure Signing and Deployment

The final stage of the deployment pipeline utilizes STM32CubeIDE to cross-compile the C firmware project. Subsequently, the STM32SigningTool authenticates the binary to satisfy secure boot constraints and the STM32CubeProgrammer transfers the artifacts to the target board via the ST-Link protocol. This entire sequence is orchestrated autonomously by the common_deploy.py script.

STM32N6570-DK_GettingStarted_PoseEstimation
Flash @ 0x70100000 (firmware) + 0x70380000 (weights)

Headless Toolchain Integration

STM32CubeIDE is the official Eclipse-based integrated development environment maintained by STMicroelectronics. Within this specific deployment architecture, it is decoupled from its graphical interface and utilized exclusively as a headless cross-compilation toolchain. The Python orchestration script invokes the compiler in the background to build the final executable, proceeding directly to the flashing sequence without requiring manual developer intervention.

A critical architectural constraint of the STM32N6570-DK is its strict secure boot protocol. A generated binary cannot execute natively upon transfer; it must first be cryptographically authenticated using the STM32SigningTool. Furthermore, the deployment workflow partitions the executable artifacts: the primary firmware and the static model weights are flashed to distinct physical addresses within the external OctoFlash memory. Understanding this partitioned memory map clarifies the necessity for dual flash commands and the physical boot switch toggling required during hardware initialization.

Build Orchestration Configuration: stmaic_STM32N6570-DK.conf

The common_deploy.py orchestration script avoids hardcoded directory paths. Instead, it dynamically resolves the workspace topology by parsing stmaic_STM32N6570-DK.conf—a JSON-formatted configuration file that dictates the exact build and flash procedure for the target hardware. The specific configuration executed for this deployment is detailed below:

    [ "", "${network_src_root}/network_atonbuf.xSPI2.raw", "copy" ],

}
Architectural Insight: The Injection Strategy
Prior to invoking the headless STM32CubeIDE compiler, the orchestration script parses the templates array to inject the auto-generated artifacts into the C firmware tree. This mechanism dictates how the ST Edge AI Core outputs (such as network.c, network_ecblobs.h and the .raw OctoFlash payload) physically interface with the compiler. Consequently, the core C project serves purely as a skeleton template; successful compilation is strictly contingent upon this dynamic injection sequence.

Automated Build & Flashing Sequence Orchestration

Upon completion of the ST Edge AI Core compilation, the common_deploy.py orchestration script parses the configuration file to execute a strict, four-stage deployment protocol. Each discrete command isolated below operates autonomously but can be executed manually for debugging purposes:

1
Artifact Injection (Template Population)
# Sourced from the templates array in the .conf definition:
cp network.c → Model/STM32N6570-DK/
cp network_ecblobs.h → Model/STM32N6570-DK/
cp network_atonbuf.xSPI2.raw → Model/STM32N6570-DK/
cp app_config.h → Application/STM32N6570-DK/Inc/
2
Headless Cross-Compilation Execution
STM32CubeIDE --launcher.suppressErrors -nosplash \
  -application org.eclipse.cdt.managedbuilder.core.headlessbuild \
  -build STM32N6570-DK_GettingStarted_PoseEstimation/Debug
# Target Artifact: Debug/STM32N6570-DK_GettingStarted_PoseEstimation.bin
3
Cryptographic Binary Signing
STM32SigningTool -s \
  -bin Debug/STM32N6570-DK_GettingStarted_PoseEstimation.bin \
  -nk -t ssbl -hv 2.3 \
  -o Debug/STM32N6570-DK_GettingStarted_PoseEstimation_signed.bin
# Mandatory authentication protocol verified by the STM32N6 FSBL upon boot
4
Target Device Flashing (Partitioned Memory Mapping)
# Transfer First-Stage Bootloader (FSBL):
STM32CubeProgrammer -c port=swd mode=HOTPLUG -hardRst \
  -w Binary/ai_fsbl.hex

# Transfer Cryptographically Signed Firmware (Target: 0x70100000):
STM32CubeProgrammer -c port=swd mode=HOTPLUG -hardRst \
  -w Debug/*_signed.bin 0x70100000

# Transfer Serialized Weight Matrix (Target OctoFlash: 0x70380000):
STM32CubeProgrammer -c port=swd mode=HOTPLUG -hardRst \
  -w Model/STM32N6570-DK/network_atonbuf.xSPI2.bin 0x70380000
Architectural Rationale: Partitioned Memory Mapping

The executable firmware (0x70100000) and the static model weights (0x70380000) are isolated into discrete memory partitions. This topology physically decouples their respective update lifecycles and satisfies the platform's secure boot constraints.

0x70100000 — Executable Firmware Partition
Contains the compiled application logic (inference loop, MIPI pipeline, spatial post-processing). This partition remains static unless the core logic is modified. It requires cryptographic signing, as the First-Stage Bootloader (FSBL) verifies its signature prior to execution.
0x70380000 — Static Weight Payload
Contains the raw, serialized INT8 model parameters (2.924 MB). Because this data is mapped strictly as read-only memory (ROM) and lacks an execution context, it bypasses the signature requirement. This permits rapid, iterative model swapping without firmware recompilation.

Hardware Boot Configuration Protocol

The STM32N6570-DK utilizes dedicated physical hardware switches to dictate the boot memory origin upon a power-on reset (POR). Improper configuration of these hardware selectors is the primary cause of silent execution failures following an ostensibly successful deployment.

System Bootloader Mode (Flashing State)
Must be configured prior to initiating the deployment operation mode. This state forces the microcontroller to boot from system memory, allowing the ST-Link interface to establish a connection and write data to the external OctoFlash.
Switch Position: RIGHT (BOOT0 = 1, BOOT1 = 1)
OctoFlash Execution Mode (Runtime State)
Must be asserted after the flashing sequence concludes, followed by a hardware power-cycle. This instructs the FSBL to boot from the OctoFlash, execute cryptographic verification and initialize the primary firmware inference loop.
Switch Position: LEFT (BOOT0 = 0, BOOT1 = 0)

4. external_memory_mgt.py → patch linker script

6. STM32CubeIDE headless → compile → .bin
7. STM32SigningTool → sign → _signed.bin
8. STM32CubeProgrammer → flash firmware @ 0x70100000
9. STM32CubeProgrammer → flash weights @ 0x70380000
